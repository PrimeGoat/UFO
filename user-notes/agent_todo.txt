--------------------------------------------------------------------------------

CONTEXT
  D:\AI\UFO\ is the UFO2 Windows UI automation project (Microsoft Research).
  The directory contains source code, YAML config files, PowerShell/batch
  scripts, Python scripts, and generated runtime artifacts (logs, screenshots,
  dumps, caches). The goal is to bring the project under version control with
  a clean separation between tracked source and generated artifacts — so that
  future changes can be tracked, accidental breakage can be detected, and the
  repo never accumulates runtime noise.

  A clean-install history must be created first from a fresh standalone
  download/extract of UFO2 and then transplanted into the existing D:\AI\UFO\
  working tree so that:
    Commit 1 = clean install baseline
    Commit 2 = current D:\AI\UFO\ setup

  OmniParser is a separate thing in .. and must be handled the same way in its
  own separate directory. Its history must likewise be:
    Commit 1 = clean install baseline
    Commit 2 = current setup

  Note: D:\AI\UFO\ should be treated as the UFO2 repo root after the Git
  metadata transplant. There is no parent git repo to worry about nesting
  inside.

STEP 1 — Clean Package Acquisition & Temporary Install Roots
  Create fresh standalone clean-install directories outside the existing setup,
  parallel in style to the current layout.

  Required:
    - Download the full UFO2 package and everything else it comes with/needs.
    - Extract/install it into a temporary clean directory named:
        ..\UFO2-clean\
    - Download/setup OmniParser separately in .. as its own temporary clean
      directory named:
        ..\OmniParser-clean\

  These temporary clean-install directories are only for generating the first
  commit history. Do not reuse the existing working directories for this step.

STEP 2 — Temporary Repository Initialization
  For EACH temporary clean-install directory (..\UFO2-clean\ and
  ..\OmniParser-clean\):
    - Run: git init
    - Do NOT make the initial commit until the appropriate .gitignore is in
      place and staged.
    - The first commit must represent the pristine clean install only.

STEP 3 — .gitignore Strategy
  Create a thorough .gitignore at the repo root (.\gitignore). Add any
  subdirectory-level .gitignore files only if clearly needed for precision.

  CRITICAL RULE:
    The .gitignore rules must NOT ignore any files that already exist in the
    freshly unpacked clean-install directories. If a file exists in
    ..\UFO2-clean\ or ..\OmniParser-clean\ immediately after unpack/setup, it
    must remain trackable and must be included in the post-install commit.

    In other words:
      - Ignore rules are for generated/runtime noise only
      - Ignore rules must be written narrowly enough that they do not match
        real files shipped by the clean install
      - If an ignore pattern would match an existing unpacked file, refine it
        with path-scoping, negation rules, or narrower patterns so that the
        existing file is NOT ignored

  MUST IGNORE (all expected runtime artifacts and generated noise, but only
  where those are not part of the unpacked clean install):
    Logs and task output:
      logs/
      *.log
      *.ndjson
      *.jsonl  (runtime task logs; NOTE: .jsonl source fixtures should be

    Crash and dump files:

      *.tmp, *.temp
      ~*, .~*

    Screenshots and captures (UFO2 saves these at runtime):
      *.png and *.jpg inside logs/ or any task output directory
      (Use path-scoped rules to avoid ignoring intentional image assets or any
      image files that are already present in the unpacked clean install)

      .cache/
      vectorstore/  (auto-generated FAISS indices; not the base vectordb/ dirs)

  After transplanting the Git metadata into the existing working tree:

    - In D:\AI\UFO\, stage all non-ignored files: git add -A

    - Create the second commit with EXACTLY this message (no extra lines):
        current setup

  This commit represents the current working customized setup layered on top of

STEP 7 — Artifact Export Script
  Write a PowerShell script at: .\scripts\export-artifacts.ps1

  The script must:

  a) Use git ls-files --others --ignored --exclude-standard to enumerate
     all files currently matching the .gitignore rules. This is the
     authoritative source — do not re-implement glob matching manually.

  b) Accept a -Mode parameter with three values:
       DryRun  → print what would happen; touch nothing (DEFAULT if omitted)
       Copy    → copy matched files to destination; originals remain in place
       Move    → move matched files to destination; removed from source

  c) Compute a timestamp-named destination directory:
       ..\UFO-artifacts\<timestamp>\
       Timestamp format: MM-DD-YY HH-MM-SSam (e.g. "03-26-26 02-14-07PM")
       Note: parent dir is D:\AI\UFO-artifacts\ (one level above the repo).

  d) Preserve original relative directory structure inside the destination.
     Example:
       Source:      .\logs\task_001\step1.png
       Destination: ..\UFO-artifacts\03-26-26 02-14-07PM\logs\task_001\step1.png

  e) Output a readable report in all modes:
       - Each file: source path → destination path
       - Summary: total file count, total size (human-readable)
       - DryRun output must be prominently labeled:
           "=== DRY RUN — no files were modified ==="

  f) Handle edge cases gracefully:
       - Destination directory already exists → create it, don't error
       - Files locked or in use → skip with a warning, continue
       - Insufficient permissions → report clearly, skip affected files
       - Empty match set → print "No artifact files found." and exit cleanly

STEP 8 — Validation

  - Run the script in DryRun mode: .\scripts\export-artifacts.ps1 -Mode DryRun
  - Review the output. Confirm:
      * No source files (.py, .yaml, .ps1, .bat, config files) are matched
      * Log files, caches, __pycache__, etc. ARE matched
      * Destination paths correctly mirror the source tree structure
  - If the dry run output is correct, run Copy mode:
      .\scripts\export-artifacts.ps1 -Mode Copy
  - Do NOT run in Move mode unless explicitly requested by the operator.

STEP 9 — Verify Final State
  After copy run completes:
  - Run git status — should show a clean working tree with only tracked files
    and expected .gitignore'd items remaining in place.
  - Confirm D:\AI\UFO-artifacts\ contains a timestamped subdirectory with
    the correct structure.
  - Confirm the history shape is correct:
      git log --oneline
        newest:   current setup

        newest:   current setup

ASSIGNMENT 5 — UFO2 MCP Server Redesign (Role-Aware, Session-Connected)
--------------------------------------------------------------------------------

CONTEXT
  The current MCP server implementation (ufo_mcp_server.py, built this session)
  is a functional but thin wrapper: it spawns UFO2 as a subprocess per task,
  pipes N\n to auto-answer the continuation prompt, and parses log files for
  results. It has no session awareness, no role communication, and no persistent
  connection to a running UFO2 instance.

  This assignment defines the redesign. The new MCP server is the authoritative
  control layer through which another agent (Claude Code, Codex, etc.) drives
  UFO2 as a subordinate, role-aware, async Windows interaction subagent.

  IMPORTANT: Before implementation begins, a preliminary research/discussion
  pass is required (documented in this assignment). The research questions must
  be answered before the implementation spec is treated as final, because UFO2's
  session model has direct implications for how the connection and declaration
  must be structured.

OUTPUT
  Primary implementation file: .\ufo_mcp_server.py  (full replacement)
  Supporting files as needed:  .\ufo_session.py, .\ufo_prompts.py, etc.
  Language: Python (required)

PRELIMINARY RESEARCH REQUIRED BEFORE IMPLEMENTATION
  The following questions must be answered first, using UFO2 source code
  inspection and any available documentation:

  1. UFO2 SESSION MODEL
     - Does UFO2 maintain any persistent session state between task invocations,
       or is each invocation entirely stateless?
     - Where does UFO2 expect its input? (stdin, CLI args, HTTP, socket?)
     - Can UFO2 be left running between tasks (persistent process), or does it
       exit after each task by design?
     - When UFO2 is started in --mode normal vs --mode follower vs --mode server:
       what is the expected I/O contract for each mode?
     - Does UFO2 have any concept of a "session prompt" or initialization message
       that is processed before the first task?
     - How does UFO2's continuation prompt work? (the "more requests?" prompt
       that the current implementation bypasses with N\n) -- is there a cleaner
       way to keep it alive between tasks without hacking stdin?

  2. STDIO COMPATIBILITY
     - If UFO2 is run in a mode that accepts stdin-based input, can a Python
       parent process write prompts to its stdin and read structured output
       from its stdout reliably?
     - Are there known issues with buffering, encoding, or blocking reads in
       UFO2's I/O that would affect a stdio-based controller?
     - Does UFO2 flush output promptly, or does it batch? This affects whether
       the MCP server can reliably detect task completion.

  3. CONTEXT AND MEMORY PERSISTENCE
     - UFO2 has an Experience Memory (RAG over past task trajectories). Does
       this persist across invocations automatically, or only when a task
       completes with SAVE_EXPERIENCE enabled?
     - Is there any per-session context (conversation history, app state) that
       survives between tasks within the same process lifetime?
     - If UFO2 exits between tasks, is all session context lost? Or does the
       RAG experience store persist on disk and reload on next start?

  These questions must be answered and their implications noted before writing
  the implementation. The answers determine whether a persistent-process model
  is viable, and how the introductory declaration must be delivered.

IMPLEMENTATION REQUIREMENTS

  Language: Python (required).

  1. STARTUP BEHAVIOR
     On MCP server startup:
       a) Check whether the UFO2 stack is already running and ready.
          Use the same health signals as Get-UFO2Status.ps1 (Assignment 3):
          OmniParser port 8010 responding, UFO2 process present.
       b) If not ready: invoke Start-UFO2Stack.ps1 -Force (non-interactive).
          Capture and monitor output. Surface any startup errors to the caller
          via structured MCP error responses -- do not silently swallow them.
       c) Do not proceed until availability check passes.
          Implement a retry loop with configurable timeout and backoff.
          Distinguish between "still loading" and "failed to start".

  2. CONNECTION TO UFO2
     After availability is confirmed:
       a) Connect to UFO2 using the most compatible stdio-capable method
          (determined by preliminary research above).
       b) The connection must support bidirectional communication with zero
          encoding or buffering issues. Use utf-8 throughout.
       c) If UFO2 requires a persistent process: manage process lifecycle
          within the MCP server (start, monitor, restart on crash).
       d) If UFO2 is inherently per-task: document why and adapt the session
          initialization flow accordingly (e.g. deliver declaration per task).

  3. SESSION INITIALIZATION — INTRODUCTORY DECLARATION
     When a new session begins (first task after startup, or after reconnect):
       a) Deliver an introductory declaration to the UFO2 agent before any
          task is submitted.
       b) The declaration is not a system prompt in the LLM sense. It is an
          explicit, natural-language initialization message that UFO2 processes
          as part of its context.
       c) The declaration must cover (preliminary recommendations -- refine
          based on research findings):

          ROLE STATEMENT
            "You are operating as an asynchronous Windows interaction subagent.
             You have been invoked by an MCP server, which was itself called as
             a tool by another AI agent. Your role is to execute Windows desktop
             tasks on that agent's behalf."

          OPERATIONAL CONTEXT
            "The agent that invoked you does not have direct access to the
             Windows desktop. You are its hands. You will receive task
             descriptions in natural language. Your job is to execute them
             reliably, verify the outcome, and report back with a structured
             result."

          COMMUNICATION CONTRACT
            "When a task is complete, report: what you did, what the outcome
             was, whether it succeeded, and any relevant observations. If you
             encounter ambiguity or a decision point that requires human
             judgment, say so explicitly rather than guessing. Do not ask
             clarifying questions for clearly specified tasks."

          BEHAVIORAL GUIDANCE
            "Be methodical. Verify each action before moving to the next.
             Prefer UIA-accessible controls over visual fallback where possible.
             If an action fails, try an alternative approach before giving up.
             Do not take destructive actions unless explicitly instructed.
             Check your actions against the original request before executing
             anything that cannot be undone."

          SECURITY POLICY (inline, natural language)
            "Before executing any task, verify that it does not violate the
             following policy: [policy text to be defined -- see Assignment 6].
             If a task appears to violate policy, refuse it and explain why.
             As a secondary check: before each individual action, verify that
             the action is consistent with the original request. If an action
             does not match what was asked, do not execute it."

       d) The declaration should be stored in a separate file (ufo_prompts.py
          or similar) so it can be updated without touching server logic.
       e) The declaration is delivered once per session. It is not re-sent
          with every task. Track session state to enforce this.

  4. TASK SUBMISSION
     For each incoming windows_task() call:
       a) Submit the task to the connected UFO2 agent.
       b) Monitor execution -- watch for completion, errors, or timeout.
       c) Parse the result from UFO2's output or log files (whichever is
          more reliable given the session model determined by research).
       d) Return a structured MCP response: status, summary, actions, errors.

  5. ERROR HANDLING
     - Startup failure: return structured error; do not leave stack half-started.
     - UFO2 crash mid-task: detect, attempt restart, report to caller.
     - Task timeout: configurable; return timeout error with partial state.
     - Policy violation detected by UFO2: surface explicitly in response.
     - Encoding errors: PYTHONIOENCODING=utf-8 must be enforced in all subprocs.

  6. TOOLS EXPOSED VIA MCP (revised from current implementation)
     Keep the same two-tool structure but expand semantics:

       windows_task(request, context?, task_id?, timeout?)
         - request:  natural language task description
         - context:  optional additional context from the calling agent

         - task_id:  optional caller-provided ID for tracking
         - timeout:  seconds before declaring task failed (default: 300)

         - status:   full stack health (process, port, health checks, GPU)
         - logs:     task log content (all recent, or specific task_id)
         - config:   effective configuration dump
         - session:  current session state (connected, declaration sent, etc.)
         - restart:  trigger a full stack restart (calls Start-UFO2Stack.ps1)

PART A: SECURITY POLICY

OBJECTIVE
  Write the natural-language security policy that is injected into the UFO2
  session declaration (Assignment 5). This policy is the first of the two-stage
  safety check: (1) does the request violate policy? (2) does each action match
  the original request?

POLICY FILE
  .\customization\security_policy.txt
  (Plain text, human-readable. Loaded by the MCP server and injected into the
   session declaration at the placeholder defined in Assignment 5.)

POLICY REQUIREMENTS
  The policy must be written in clear, unambiguous natural language that an LLM
  can reason against. It must cover:

  1. DESTRUCTIVE ACTIONS
     - No deletion of files or directories unless explicitly named in the request.
     - No uninstalling of software unless the software is explicitly named.
     - No formatting or wiping of drives under any circumstances.
     - No modification of system files, registry, or security settings.
     - No termination of processes unless explicitly requested and named.

  2. DATA EXFILTRATION
     - Do not read, copy, transmit, or expose files containing credentials,
       API keys, passwords, or private keys.
     - Do not send any data to external services or URLs not explicitly named
       in the request.
     - Do not read browser-saved passwords, autofill data, or stored tokens.

  3. IRREVERSIBLE OPERATIONS
     - Before any action that cannot be undone: verify it matches the request.
     - If uncertain whether an action is reversible: treat it as irreversible.

  4. SCOPE CONTAINMENT
     - Work only within the applications and files explicitly mentioned.
     - Do not open, read, or modify files not related to the task.
     - Do not install software not explicitly requested.

  5. NETWORK OPERATIONS
     - Do not initiate network connections not required for the stated task.
     - Do not submit forms or post data to external URLs without explicit
       instruction.

  POLICY VERSIONING
  Include a version header and date in the file so the MCP server can log which
  policy version was active for each task.

PART B: QA PAIR FAISS FILTERING

BACKGROUND
  UFO2 loads QA pairs from QA_PAIR_FILE into the agent prompt for context.
  Currently it loads the last N lines (QA_PAIR_NUM=30) with no relevance
  filtering -- this is wasteful and injects irrelevant context.

OBJECTIVE
  Build a lightweight FAISS-based filter that prepares a per-request subset of
  QA pairs, selecting only the ones most relevant to the incoming task.

IMPLEMENTATION LOCATION
  .\ufo_qa_filter.py   (standalone module, imported by ufo_mcp_server.py)

REQUIREMENTS
  1. INDEX BUILDING
     - On startup (or on first call), embed all QA pairs from QA_PAIR_FILE
       using sentence embeddings (all-MiniLM-L6-v2 already present).
     - Build a FAISS flat index (IndexFlatL2 or IndexFlatIP).
     - Cache the index in memory. Rebuild when QA_PAIR_FILE mtime changes.

  2. QUERY
     - Given the incoming task string, embed it with the same model.
     - Retrieve top-K most similar QA pairs (K = QA_PAIR_NUM = 30).
     - Return as a list of dicts matching the existing QA pair format.

  3. INTEGRATION
     - ufo_mcp_server.py calls ufo_qa_filter.get_relevant_qa(task, k=30)
       before submitting the task to UFO2.
     - The filtered QA pairs are written to a temp file that overrides
       QA_PAIR_FILE for the duration of the task.
     - Fallback: if FAISS is not installed or index build fails, fall back to
       existing last-N behavior with a warning.

  4. COST AWARENESS
     - Embedding model runs locally (no API calls).
     - FAISS is CPU-only (no GPU required at this scale).
     - Index rebuild must be fast enough not to block task submission.

  5. DEPENDENCIES
     - faiss-cpu (pip install faiss-cpu)
     - sentence-transformers (already present in ufo conda env)
